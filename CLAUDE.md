# CLAUDE.md

本文件供 Claude Code 在本仓库中工作时自动读取，用于快速了解项目背景、技术约束和开发规范。完整需求请参阅仓库根目录下的《TRX兑换能量小程序-需求设计文档.md》，本文件不重复其细节，只提炼开发时必须遵守的规则。

## 项目是什么

一个 TRON（波场）"以TRX换能量"服务的展示与统计站点。客户向指定地址转账 1 TRX 后，由**第三方能量托管平台**自动为其分配能量——这部分不属于本项目。本项目只做三件事：前台展示收款地址并引导复制、后台可配置站点信息、后台统计复制次数与转账次数并计算转化率。

## 严禁扩大范围（务必遵守）

- **不要实现任何能量委托/下发逻辑**，不要引入 TronWeb 私钥签名、Stake/DelegateResource 相关代码。收款地址已由外部平台托管，本项目对链上资源操作没有任何权限，也不应该有。
- **不要实现 TRX 转账/退款逻辑**，本项目不接触任何私钥，只做只读的链上数据查询（TronGrid API 查交易记录）。
- 涉及链上交互的代码只允许出现在 `cron/fetch_transfers.php` 中，且只能是只读的 GET 请求。
- 如果任务描述看起来要求实现能量分配或转账，先停下来向用户确认，不要假设范围扩大了。

## 技术栈约束

- PHP 8.x，原生实现，不引入 Laravel/Symfony 等框架。
- 数据库为 MySQL（5.7+/8.0），所有 SQL 通过 PDO 预处理语句执行，**禁止字符串拼接 SQL**。
- 前端为原生 HTML/CSS/JS，移动端优先响应式布局；富文本编辑器用 Quill.js；二维码用 qrcode.js 之类的前端库纯客户端生成，不在后端存储二维码图片。
- 不引入 Composer 依赖管理框架级别的包，保持轻量；如需引入第三方库（如 DOMPurify 做 XSS 过滤），优先选择免费、无需 API Key、可自托管的方案。

## 目录结构

本地开发命令 `php -S localhost:8000 -t public` 要求 **`public/` 就是 Web 文档根目录**，因此所有需要通过 HTTP 访问的目录（后台、接口、安装向导、上传文件）都物理上位于 `public/` 之下；不需要对外暴露的目录留在项目根目录：

```
public/             文档根目录（生产环境 Nginx/Apache 的 root 直接指向这里）
  index.php         前台入口
  assets/           前台静态资源（css/js/vendor）
  admin/            后台管理页面与逻辑
  api/              前台接口（如 /api/copy.php）
  install/          安装向导，安装完成后建议提示用户删除或重命名
  uploads/          Logo等上传文件，需要可被浏览器直接访问

includes/   公共类库（Database 单例、Auth、JSON输出封装等），项目根目录，不对外暴露
config/     database.php 等配置文件，项目根目录，不对外暴露（.htaccess 兜底禁止HTTP直接访问）
cron/       定时任务脚本（fetch_transfers.php），项目根目录，不对外暴露
cli/        命令行应急工具（如 reset_password.php），项目根目录，不对外暴露HTTP入口
logs/       日志文件，项目根目录，不对外暴露
.env        收款地址二次确认密码等不放数据库的敏感配置，项目根目录，不对外暴露
```

新增文件时按此结构归位：**凡是浏览器要直接访问的东西放进 `public/` 对应子目录；凡是只被 PHP 内部 include/CLI 调用的东西留在项目根目录**，不要把业务逻辑散落到别处。`includes/`、`config/`、`cron/`、`cli/`、`logs/` 均已放置 `.htaccess`（`Require all denied`）作为纵深防御，但主要防线是它们本就不在文档根目录之下。

## 安全红线（代码评审时重点检查）

1. 站点必须假设运行在 HTTPS 之下；不要在代码里写死 `http://` 的绝对链接。
2. 收款地址是资金安全的核心：前台展示的地址只能来自服务端渲染的 `config` 表数据，绝不接受任何 URL 参数、Cookie 或客户端输入覆盖显示的地址。
3. 后台保存收款地址（`/admin/config/save.php` 涉及地址变更时）必须要求当前登录密码二次确认，校验失败则拒绝写入。
4. 所有数据库读写走 PDO 预处理语句；用户可控内容（尤其是富文本"规则说明"）输出到前台前必须做 XSS 过滤。
5. `install/` 目录下的每个脚本，执行任何操作前先检查 `install.lock` 是否存在，存在则直接拒绝并跳转提示页。
6. `config/database.php`、`data/`、`logs/` 等目录需要有对应的 Nginx/Apache 规则禁止直接访问，写代码时不要假设这些目录是公开可读的。
7. Logo/图片上传要校验 MIME 类型和大小，并以随机文件名重命名存储，不信任用户提供的原始文件名。

## 关键设计决定（不要重新发明）

- 复制次数用 `copy_events` 明细表 + `SELECT COUNT(*) ... WHERE DATE(created_at)=?` 聚合，不要额外维护每日计数缓存表——数据量级不需要。
- 转账记录去重靠 `transfer_records.tx_hash` 的唯一索引，写入时捕获唯一键冲突并忽略即可，不要在业务代码里先查询是否存在再插入（有竞态风险）。
- 转账次数统计按 `tx_timestamp`（链上时间）而不是 `fetched_at`（抓取时间）来判断"今天/昨天"。
- 转化率是参考性指标，前端和后台文案里不要暗示这是精确的逐笔转化追踪。
- 金额判断用 `>= 1 TRX`，不要做精确等于匹配（用户可能多转）。

## 本地开发 / 常用命令

项目无构建步骤，PHP 直接运行即可：

```bash
# 启动本地开发服务器（以 public/ 为文档根目录）
php -S localhost:8000 -t public

# 手动跑一次转账记录抓取（调试用，正常应由crontab调度）
php cron/fetch_transfers.php

# 应急重置管理员密码
php cli/reset_password.php --username=admin --password=NewPassw0rd
```

数据库表结构以需求文档第六章的 `CREATE TABLE` 语句为准；安装向导（`install/`）应在首次访问时自动执行这些建表语句，不需要额外维护单独的 migration 文件，除非后续项目复杂度明显上升。
